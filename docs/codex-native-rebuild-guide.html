Codex-native rebuild guide

Claude와 Codex를 분리하고, Codex 하네스를 별도 설계한다

이 문서는 Claude 하네스와 Codex 하네스를 서로 독립된 active harness로 분류하고, Codex 환경에만 적용되는 skill, agent, hook, AGENTS 체계를 3번 검토해 구축하는 방법을 설명한다.

목차 핵심 결정 현재 기준 상태 하네스 경계 3-Pass 검토 변환 매핑 자산별 처리 스킬 호출 방식 Agent 한도와 Goal 검증과 인코딩 작업 흐름 위험과 대응

핵심 결정

Claude 하네스는 Claude-native active harness로 계속 유지하고, Codex 하네스는 Codex-native active harness로 별도 운영한다. 이 저장소가 실제로 설정하는 범위는 Codex 환경뿐이다.

결론: clone이나 기본 실행만으로 사용자의 홈 환경을 수정하지 않는다. 실제 적용은 --apply를 명시했을 때만 ~/.codex와 ~/.agents/skills에 수행한다.

왜 단순 복사가 부족한가

  • Claude slash command는 Codex skill trigger와 같은 런타임이 아니다.
  • Claude의 AskUserQuestion, Task tool, TodoWrite는 Codex에서 같은 이름으로 실행되지 않는다.
  • allowed-tools, argument-hint, user_invocable 같은 frontmatter는 Codex 권한이나 호출 규칙이 아니다.
  • 많은 원본 스킬이 skill.md 소문자 entrypoint를 사용한다. Codex에서는 SKILL.md로 안정화해야 한다.
  • macOS에서는 대소문자 비민감 파일시스템 때문에 SKILL.md처럼 읽혀도 실제 파일명이 skill.md일 수 있다.

인벤토리 도구 기준으로 Claude-native와 Codex-native 하네스는 다음 상태다. 이 숫자는 재구축 진행 상황을 측정하는 baseline이다.

0 Codex skills with Claude runtime markers
항목 상태 의미

하네스 경계

재구축은 한 방향 복사가 아니다. Claude와 Codex를 독립 환경으로 분류하고, Codex에서 필요한 구현만 Codex-native 형태로 작성하는 과정이다.

1. Claude-native active harness

~/.claude 아래의 skills, commands, agents, hooks, scripts, settings, rules는 Claude 환경에서 계속 사용한다.

2. Review gate

Claude 의도, Codex 매핑, 누락/충돌을 3번 검토한다. 검토가 끝나기 전에는 Codex active harness를 고치지 않는다.

3. Codex-native active harness

~/.agents/skills, ~/.codex/AGENTS.md, agents, hooks, scripts로 별도 구현한다.

Claude-native active harness
  ~/.claude/skills
  ~/.claude/commands
  ~/.claude/agents
  ~/.claude/hooks
  ~/.claude/scripts
  ~/.claude/settings*

Codex-native active harness
  ~/.agents/skills
  ~/.codex/AGENTS.md
  ~/.codex/agents
  ~/.codex/hooks.json
  ~/.codex/config.toml
  ~/.codex/scripts

Project control plane
  tools/*
  docs/*
운영 규칙: Claude 경로에는 쓰지 않는다. 이 저장소의 적용 도구는 Codex 경로에만 쓰며, 기본 실행은 dry-run이다. 실제 적용은 --apply를 명시해야 한다.

3-Pass 검토

사용자의 스킬은 자산급이므로 한 번 읽고 변환하지 않는다. 각 스킬 또는 설정 묶음은 최소 세 관점에서 확인한다.

Pass 목적 질문 산출물
Pass 1
원본 의도 추출		 Claude Markdown을 그대로 읽고 원래 의도를 이해한다. 무엇을 자동화하려 했나? 언제 트리거되나? 어떤 산출물을 기대하나? 트리거, 목적, 워크플로우, 도구, 확인 지점, 산출물 요약
Pass 2
Codex 매핑		 원본 의도를 Codex 기능으로 옮길 수 있는지 판단한다. skill로 둘 것인가? custom agent인가? hook인가? script인가? 수동 절차인가? Codex primitive 매핑표, 미지원 의미, fallback 전략
Pass 3
누락/충돌 검토		 빠뜨린 규칙, 잘못 이해한 표현, 시스템 지침 충돌을 다시 찾는다. Claude 전용 표현이 실행 지시로 남았나? 원본 철학이 손실됐나? 권한 모델이 오해됐나? 최종 rewrite checklist, residual risk, 보류 항목
Pass 3 실패 시 되돌아가기 문제가 발견되면 바로 쓰지 않는다. Pass 1 또는 Pass 2로 되돌아가 의도와 매핑을 다시 확인한다.
검토 결과를 문서화 변환 결과만 남기지 않는다. 왜 그렇게 바꿨는지 문서에 남겨야 다음 세션이 이어받을 수 있다.
불명확하면 보류 미지원 의미나 Claude 의도가 불명확한 경우 Codex active harness에 억지로 반영하지 않고 미확정 항목으로 둔다.

변환 매핑

아래 매핑은 기계적인 문자열 치환이 아니라 의미 변환이다. 같은 단어라도 Codex에서 실행 의미가 다르면 행동 지침으로 낮추거나 다른 primitive로 옮긴다.

Claude-native 요소 Codex-native 구현 왜 이렇게 바꾸는가
CLAUDE.md AGENTS.md Codex는 AGENTS 지침 체계를 사용한다. 프로젝트별 지침도 같은 이름을 따른다.
~/.claude/skills ~/.agents/skills 사용자 스킬의 공식 위치를 Codex 기준으로 고정한다.
skill.md SKILL.md Codex 인덱싱 안정성을 위해 entrypoint 대문자를 표준으로 둔다.
/pdca, /think-teams source-command-pdca 실행, think-teams 실행 같은 자연어 호출 Claude slash command 런타임은 Codex skill 호출과 다르다.
$ARGUMENTS 현재 사용자 요청 또는 필요한 경우 짧은 질문 Codex에서 자동 치환되는 command argument가 아니다.
AskUserQuestion 필요한 경우 사용자에게 직접 질문 도구 호출 강제 대신 Codex 대화 흐름에 맞게 확인 지점을 둔다.
TodoWrite update_plan Codex의 계획/진행 상태 도구가 역할을 대체한다.
Task tool, Agent tool spawn_agent 정책 또는 순차 fallback Codex subagent는 사용 조건, 권한, 위임 정책이 다르다.
tools allowlist developer instruction의 행동 가이드 Codex custom agent TOML의 권한 경계가 아니므로 보안 모델로 오해하면 안 된다.
PreCompact prompt hook Stop checkpoint, 수동 점검 skill, 문서화 절차 Codex hook lifecycle과 1:1 대응하지 않는다.

자산별 처리 방식

1. Skills

가장 중요한 자산이다. Claude skill은 Claude 환경에서 유지하고, Codex skill은 Codex 환경에 맞게 다시 쓴다.

  • SKILL.md 대문자 entrypoint를 만든다.
  • frontmatter는 최소 name, description을 둔다.
  • description에는 한국어와 영어 트리거를 함께 넣는다.
  • 본문의 Claude 전용 도구명은 Codex 행동 규칙으로 재작성한다.
  • 긴 참조 자료는 references/로 분리해 progressive disclosure를 유지한다.

2. Commands

Claude command는 Codex에서 slash command로 직접 실행된다고 가정하지 않는다. 대신 source-command-* skill로 의미를 보존한다. 

Claude:
  ~/.claude/commands/pdca.md
  invocation: /pdca <target>

Codex:
  ~/.agents/skills/source-command-pdca/SKILL.md
  invocation: "source-command-pdca 실행", "PDCA Gap 검증해줘"

3. Agents

Claude agent Markdown은 Codex custom agent TOML로 변환한다. 읽기 전용 계열과 쓰기 가능 계열을 나누고, Claude tool allowlist는 권한이 아니라 의도 설명으로 남긴다.

Read-only 계열

ce-reviewer, codebase-explorer, harness-evaluator, infra-auditor

Workspace-write 계열

thoughts-writer는 .thoughts/ 생성 목적에 한해 쓰기 가능으로 둔다.

4. Hooks와 Scripts

Codex에서 의미가 명확한 command hook만 활성화한다. prompt hook이나 lifecycle이 다른 hook은 자동 실행보다 수동 점검이나 skill로 보존한다.

  • PreToolUse: 파괴적 명령, 루프, 위험 패턴 사전 검사
  • PostToolUse: 실행 후 상태 기록
  • SessionStart: 프로젝트 상태 요약
  • Stop: checkpoint 저장

UserPromptSubmit checkpoint 알림은 비활성화한다. 이 이벤트는 출력이 있을 때 JSON-safe 형식이 필요하며, plain-text 알림은 invalid user prompt submit JSON output 오류를 만들 수 있다.

5. Rules와 AGENTS

글로벌 지침은 짧게 유지하고, 상세 규칙은 ~/.codex/rules와 ~/.agents/skills/_core로 분리한다. 이것이 CE의 progressive disclosure 방식이다.

스킬 호출 방식

Claude에서 가져온 스킬이 안 되는 것처럼 보이는 주된 이유는 파일 이전 실패가 아니라 호출 방식과 인덱싱 차이다.

기대 Codex에서의 실제 방식 예시
Claude처럼 /pdca 입력 Codex skill name 또는 자연어 트리거 사용 source-command-pdca 실행해서 설계와 구현 gap 봐줘
이전 직후 새 스킬 자동 인식 현재 세션의 skill index가 갱신되지 않을 수 있음 새 Codex 세션을 열어 목록을 갱신
skill.md도 동일하게 인식 SKILL.md로 정규화하는 것이 안전 think-teams/SKILL.md
Claude tool 이름 그대로 실행 Codex 도구와 대화 정책에 맞게 재작성 필요 TodoWrite 대신 update_plan
사용자 관점의 호출 원칙: slash command 이름을 그대로 쓰기보다 “스킬 이름 + 목적”으로 말하는 것이 안정적이다. 예: think-teams로 이 전략을 다각도 분석해줘.

Agent 한도와 Goal

현재 custom agent 정의는 5개지만, 이것은 역할 정의 개수다. Codex 런타임의 동시 agent thread 운영 한도는 별도의 설정인 agents.max_threads로 제어한다.

항목 설정 의미
Custom agent definitions 5개 TOML ce-reviewer, codebase-explorer, harness-evaluator, infra-auditor, thoughts-writer
Agent thread cap agents.max_threads = 128 동시에 열 수 있는 agent thread 상한. 기본 fan-out 수가 아니라 운영 한도다.
Goal feature features.goals = true 로컬 Codex CLI에서 under development로 확인된 장기 목표/재개 흐름 실험 기능이다.
운영 원칙: 128은 “항상 128개를 spawn한다”는 뜻이 아니다. 사용자가 명시적으로 병렬 위임을 요청하고, 작업을 독립적으로 쪼갤 수 있을 때만 필요한 만큼 사용한다.

검증과 인코딩

이 하네스는 한글과 긴 Markdown이 많다. 그래서 인코딩 검증은 부가 절차가 아니라 핵심 안전장치다.

적용 전
  • 백업 생성
  • SHA-256 manifest 생성
  • UTF-8 strict read
  • U+FFFD 검사
적용 후
  • JSON/TOML parse
  • Python compile
  • Shell syntax check
  • Hook simulation
python3 tools/codex_harness_inventory.py --output docs/claude-corpus-inventory.md
python3 tools/codex_native_harness_migrate.py
python3 tools/codex_harness_verify.py --markdown-output docs/harness-verification-report.md --json-output docs/harness-verification-report.json
bash ~/.agents/skills/infra-audit/scripts/infra-audit.sh --quick
중단 조건: Codex active harness에 invalid UTF-8 또는 U+FFFD가 남으면 적용을 중단한다. Claude 파일에 이미 있던 깨짐은 Claude 환경을 수정하지 않고 Codex active harness에서만 문맥 복구한다.
GitHub 전 게이트: 외부 GitHub 하네스 후보를 읽거나 push를 준비하기 전에 docs/harness-verification-report.md의 FAIL이 0개인지 확인한다. WARN은 Claude reference-only 깨짐, Codex validator 크기 경고, Git 초기화 전 상태처럼 원인과 허용 범위가 문서화된 경우에만 통과로 본다.

작업 흐름

앞으로의 작업은 배치 단위로 진행한다. 배치 하나는 관련 스킬 묶음, 관련 agent, 관련 hook 또는 문서 묶음을 뜻한다.

  1. 인벤토리를 실행해 현재 상태를 확인한다.
  2. 이번 배치에서 참고할 Claude-native 파일과 수정할 Codex-native 파일을 고른다.
  3. Pass 1, Pass 2, Pass 3 검토를 기록한다.
  4. Codex active harness를 재작성한다.
  5. 관련 문서를 업데이트한다.
  6. dry-run, 인벤토리, 상세 검증 게이트를 실행한다.
  7. 새 Codex 세션에서 trigger 동작을 확인한다.
  8. GitHub에 올릴 변경 묶음을 정리한다.

추천 우선순위

우선순위 대상 이유
P0 source-command-* Batch 1에서 Codex-native skill로 재작성 완료. 이후 호출 예시와 실제 trigger 품질을 새 세션에서 확인한다.
P1 think-*, what*, cynefin, ooda 구조 정규화는 완료. 다음 단계는 3-pass로 의미와 진행 방식을 더 매끄럽게 다듬는 것이다.
P1 agent-teams-*, agent-architect Codex subagent 정책과 충돌하지 않게 재설계해야 한다.
P2 infra-audit, security-audit, QA 계열 재구축 품질을 검사하는 기준점이다.
P3 문서, 오피스, 배포, 웹 디자인 보조 스킬 핵심 하네스 안정화 이후 trigger와 산출물 중심으로 정규화한다.

위험과 대응

위험 왜 중요한가 대응
원본 의미 손실 스킬은 사용자의 작업 자산이다. 기능만 옮기면 판단 기준이 사라질 수 있다. Claude-native 하네스와 Codex-native 하네스를 분리하고 3-pass 검토를 기록한다.
Codex 권한 모델 오해 Claude allowlist를 Codex 권한처럼 믿으면 안전 경계가 흐려진다. allowlist는 행동 가이드로만 남기고 실제 권한은 Codex sandbox와 approval을 따른다.
스킬 인덱스 미갱신 파일을 고쳐도 현재 세션에서 바로 보이지 않을 수 있다. 새 세션에서 trigger를 확인하고, 문서에 호출 예시를 남긴다.
인코딩 깨짐 한글 스킬과 긴 Markdown에서 깨짐이 나면 자산 가치가 크게 훼손된다. UTF-8 strict, U+FFFD, manifest 검증을 적용 전후로 실행한다.
과도한 자동 변환 문맥을 이해하지 못한 치환은 더 위험한 하네스를 만든다. 자동화는 인벤토리와 검증에 쓰고, 의미 변환은 batch별로 검토한다.
