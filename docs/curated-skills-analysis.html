Codex Harness Asset

Curated Skills 분석 카탈로그

이번 섹션에서 만든 10개 Codex-native 스킬을 목적, 트리거, 질문 포인트, 실행 흐름, 산출물, 주의점 기준으로 분석한 운영 문서다. `$what`은 새로 만든 스킬이 아니라 행동 계약 테스트의 골든 레퍼런스로 사용한다.

목차 요약 라우팅 지도 스킬별 분석 검증 계열 비교 행동 계약 운영 기준

요약

이 10개 스킬은 계획을 날카롭게 만들고, 제품 요구사항으로 정리하고, 테스트 주도 구현과 진단을 수행하고, 실제 브라우저 검증과 기록까지 이어지도록 만든 작은 운영 체계다.

10 curated Codex-native skills
35 behavior contract scenarios
3 main operating bands: shape, build, verify
0 behavior contract failures at creation time
핵심 판단

이 스킬들은 “실행 지시 묶음”이 아니라 Codex가 언제 질문하고, 언제 멈추고, 어떤 도구를 우선해야 하는지 명시한 행동 계약이다. 따라서 새 스킬을 가져올 때도 파일 형식 검증만 보지 않고, tools/codex_skill_contract_verify.py로 질문/중단/도구 우선순위까지 검증한다.

라우팅 지도

사용자의 요청은 보통 세 구간 중 하나로 들어온다. 모호한 계획은 shape 계열, 구현과 결함은 build/fix 계열, 실제 제품 증명과 반복 검증은 verify/record 계열로 보낸다.

Shape

grill-me는 질문으로 결정을 확정하고, to-prd는 확정된 맥락을 제품 요구사항으로 만든다.

Build / Fix

tdd는 한 행동씩 구현하고, diagnose는 재현 루프를 먼저 만든다. improve-codebase-architecture는 구조 후보를 찾는다.

Verify / Record

live-verify-loop, ultradetail-walk, ultradetail-loop, harness-loop, cantos-write가 실제 증명과 기록을 담당한다.

상황 우선 스킬 이유
계획은 있는데 결정이 흐릿하다 grill-me 한 번에 하나의 결정 질문으로 불확실성을 줄인다.
대화나 기능 구상을 PRD로 만들고 싶다 to-prd 인터뷰를 기본값으로 하지 않고, 이미 있는 맥락을 제품 문서로 변환한다.
테스트를 먼저 두고 구현해야 한다 tdd RED, GREEN, REFACTOR를 한 행동 단위로 반복한다.
원인 모를 실패나 성능 회귀가 있다 diagnose 수정 전에 재현 가능한 pass/fail 루프와 가설 경쟁을 만든다.
아키텍처 개선 후보를 찾고 싶다 improve-codebase-architecture 바로 리팩터링하지 않고 후보, 마찰, testability impact를 먼저 제시한다.
실제 브라우저에서 통과할 때까지 반복 검증해야 한다 live-verify-loop code green과 live green을 분리하고 Browser Use 중심으로 증명한다.
페이지/플로우의 버튼과 persona를 한 번 전부 걸어야 한다 ultradetail-walk 한 번의 exhaustive discovery pass로 결함을 찾는다.
출시 전까지 walk, fix, re-walk를 반복해야 한다 ultradetail-loop 각 라운드마다 전체 walk를 다시 수행하는 고비용 release gate다.
명시적으로 에이전트/오케스트레이션을 요청했다 harness-loop subagent 사용 권한, ownership, quality gate를 통제한다.
Cantos에 ADR, DDR, journal 기록이 필요하다 cantos-write MCP 연결 여부를 확인하고 remote write와 local fallback을 구분한다.

스킬별 분석

각 분석은 운영자가 실제로 라우팅할 때 보는 기준이다. “무엇을 하는가”보다 더 중요한 것은 “언제 질문해야 하고, 무엇을 하면 안 되는가”다.

1. grill-me

모호한 계획을 실행 가능한 결정 상태로 바꾸는 질문형 스킬이다.

Shape / planning interview
역할

한 번에 하나의 load-bearing decision을 질문한다. 계획 검증, 제품 아이디어 정리, 아키텍처 선택, 애매한 구현 요청에 적합하다.

사용자에게 물어볼 것
  • 성공 기준과 non-goal
  • 사용자/이해관계자의 행동
  • 데이터 모델, 외부 의존성, rollback
기대 산출물

Goal, confirmed decisions, constraints, risks, first implementation move를 담은 decision brief.

주의점

질문으로 해결할 수 있는 상황에서 바로 구현하지 않는다. repo에서 답을 알 수 있으면 먼저 rg와 파일 읽기로 확인한다.

2. to-prd

대화, repo 맥락, 기능 논의를 제품 요구사항 문서로 바꾸는 스킬이다.

Shape / product document
역할

이미 알려진 맥락을 PRD로 합성한다. 문제, 솔루션, 사용자 스토리, 구현 결정, 테스트 결정, out of scope를 분리한다.

사용자에게 물어볼 것

기본적으로 인터뷰하지 않는다. 다만 tracker 대상, 제품 결정, 공개 범위처럼 PRD가 크게 달라질 결정이 비어 있으면 한 가지 질문을 한다.

기대 산출물

제품 관점 PRD 또는 issue tracker용 본문. confirmed decision과 inferred decision을 구분해야 한다.

주의점

불안정한 파일 경로나 비밀 정보, private transcript를 PRD에 넣지 않는다. issue를 만들 때는 target repo/project를 확인한다.

3. tdd

한 번에 하나의 관찰 가능한 행동을 test-first로 구현하는 스킬이다.

Build / feedback loop
역할

RED, GREEN, REFACTOR를 vertical slice로 반복한다. 모든 테스트를 먼저 상상해서 쓰는 방식이 아니라, 한 행동의 실패와 성공을 차례로 확인한다.

사용자에게 물어볼 것

public interface가 불명확할 때만 묻는다. 테스트 runner를 찾을 수 없거나, 의미 있는 테스트가 production credential을 요구하면 멈춘다.

기대 산출물

실패하는 테스트, 최소 구현, green 상태, 필요한 리팩터링, 마지막 broader test command 결과.

주의점

내부 구조를 고정하는 brittle test를 피하고, refactor는 green 이후에만 수행한다. 사용자가 테스트 없는 구현을 명시하면 TDD 경로를 중단한다.

4. diagnose

버그와 성능 회귀를 수정 전에 재현 루프와 가설로 좁히는 스킬이다.

Fix / diagnosis
역할

“보고 고치기”를 금지하고, pass/fail loop를 만든 뒤 원인을 좁힌다. hard failure, intermittent bug, performance regression에 적합하다.

사용자에게 물어볼 것

재현 루프를 만들 수 없을 때 logs, HAR, screen recording, fixture, 환경 접근 같은 captured artifact를 요청한다.

기대 산출물

재현 방법, 3-5개 ranked hypotheses, probe 결과, 최소 수정, regression test, confirmed cause, prevention opportunity.

주의점

여러 변수를 동시에 바꾸지 않는다. 임시 로그는 [DEBUG-...] 형태로 표시하고 완료 전에 제거한다.

5. improve-codebase-architecture

구조 개선 후보를 찾고, testability와 locality 관점에서 우선순위를 잡는 스킬이다.

Architecture / candidate discovery
역할

얕은 wrapper, tangled seam, 테스트 불가능한 흐름, AI가 탐색하기 어려운 구조를 찾아 후보로 제시한다.

사용자에게 물어볼 것

후보를 제시한 뒤 어떤 후보를 더 탐색할지 묻는다. full interface나 대규모 refactor는 사용자가 후보를 고른 뒤로 미룬다.

기대 산출물

candidate name, files/modules, problem, solution direction, benefits, testability impact, ADR/context conflicts.

주의점

모든 abstraction을 좋은 것으로 취급하지 않는다. ADR을 가볍게 뒤집지 않고, 안정된 architectural reason이 있으면 기록을 제안한다.

6. live-verify-loop

코드 검증과 실제 브라우저 검증 사이의 gap을 반복적으로 닫는 스킬이다.

Verify / repeated live proof
역할

code green != live green을 기본 원칙으로 삼는다. 서버 health, live browser, data/API, code quality를 한 라운드로 묶는다.

사용자에게 물어볼 것
  • target scope
  • verification mode
  • termination criteria
  • tool plan
  • round limits
기대 산출물

각 라운드의 checks, fixed defects, browser/data evidence, criteria still failing, next action.

주의점

Browser Use가 live evidence를 만들 수 있으면 Playwright MCP를 요구하지 않는다. screenshot만으로 통과를 선언하지 않는다.

7. harness-loop

명시적으로 orchestration이 필요한 작업을 quality gate와 함께 진행하는 스킬이다.

Orchestration / quality gates
역할

build, fix, audit, analyze, cleanup으로 작업을 분류하고, 필요한 경우에만 subagent를 쓰며, 결과를 검증한다.

사용자에게 물어볼 것

사용자가 delegation, agents, parallel work를 명시하지 않았는데 subagent가 꼭 필요해진 경우 중단하고 확인한다.

기대 산출물

selected mode, agents/local roles used, files changed, checks run, unresolved issues, next recommended round.

주의점

worker ownership은 겹치지 않게 하고, UI 변경은 Browser Use 우선으로 실증한다. static analysis만으로 UI proof를 대체하지 않는다.

8. cantos-write

Cantos MCP에 ADR, DDR, journal, visual evidence를 남기는 기록 스킬이다.

Record / durable knowledge
역할

Cantos MCP가 연결되어 있으면 MCP-first로 기록하고, 연결되어 있지 않으면 local Markdown fallback을 명확히 구분한다.

사용자에게 물어볼 것

project registration이나 첫 remote write는 명시적 승인을 묻는다. MCP가 없을 때 local draft를 만들지 여부도 확인한다.

기대 산출물

ADR, DDR, journal 또는 project registration 결과. visual record는 route, viewport, evidence, decision, impact를 포함한다.

주의점

remote write가 안 됐는데 된 것처럼 말하지 않는다. Cantos 기록은 테스트나 live verification의 대체물이 아니다.

9. ultradetail-walk

한 번의 깊은 persona walk로 route, element, adversarial defect를 찾는 스킬이다.

Verify / one-shot exhaustive walk
역할

rendered once와 실제 persona walk를 구분한다. 모든 relevant button, link, input, menu, dialog, upload control을 walk한다.

사용자에게 물어볼 것

target routes, environment, user roles, test accounts, scope size, Cantos 기록 여부를 preflight에서 확인한다.

기대 산출물

personas walked, route and element coverage, findings table, severity, reproduction, evidence, suggested owner, live-verify handoff 후보.

주의점

결함을 고치는 loop가 아니다. 넓은 범위에서는 defect category와 persona를 2-4개 옵션으로 제시하고 조정받는다.

10. ultradetail-loop

walk, fix, re-walk를 라운드 단위로 반복하는 고비용 release gate 스킬이다.

Verify / release convergence
역할

ultradetail-walk의 깊이와 live-verify-loop의 반복성을 결합한다. 출시 전 critical flow와 multi-role system에 적합하다.

사용자에게 물어볼 것

target scope, roles, destructive-flow boundary, hard limits, stagnation limit, per-round evidence expectation을 확인한다.

기대 산출물

round별 personas, routes, findings, fixes, checks rerun, remaining defects, stop/continue decision.

주의점

간단한 failing test나 작은 PR에는 과하다. 같은 defect가 세 번 살아남으면 재설계나 범위 축소를 검토한다.

검증 계열 비교

live-verify-loop, ultradetail-walk, ultradetail-loop는 비슷해 보이지만 비용과 목적이 다르다. 잘못 라우팅하면 너무 얕거나 너무 비싼 검증이 된다.

스킬 검증 깊이 반복성 주요 질문 언제 피할까
live-verify-loop critical journey 중심 반복 fix/verify 무엇이 통과되면 멈출까? 모든 요소를 한 번씩 다 눌러야 할 때
ultradetail-walk persona와 interactive element exhaustive pass one-shot discovery 어떤 persona와 category로 걸을까? 결함 수정까지 반복해야 할 때
ultradetail-loop 매 라운드 exhaustive walk 반복 walk/fix/walk 몇 라운드와 어떤 stop condition을 둘까? 작은 버그, 단일 페이지 smoke check, 비용이 중요한 작업

행동 계약

각 스킬은 사람이 읽기 좋은 문서이면서, 동시에 모델이 지켜야 할 계약이다. 이번 작업에서는 이 계약을 별도 검증기로 고정했다.

검사 항목 의미 실패 시 영향
Trigger terms 언제 스킬을 호출해야 하는지 frontmatter와 본문에 드러나는가 스킬이 필요한 상황에서 로딩되지 않거나 과잉 호출된다.
Question / confirmation contract 사용자에게 무엇을 물어야 하는지와 언제 멈춰야 하는지가 명확한가 모호한 상태에서 작업이 진행되거나 `$what`류 확인 루프가 깨진다.
Tool priority Terminal, Browser Use, Computer Use, Playwright/scripts, MCP의 우선순위가 맞는가 Codex Desktop에 맞지 않는 Playwright/MCP 의존으로 흐를 수 있다.
Stop conditions credentials, destructive action, missing tooling, repeated failure 시 멈추는가 위험한 작업이나 비용 큰 루프가 무제한 진행될 수 있다.
Runtime residue Claude 전용 runtime marker가 Codex 실행 지시로 남아 있지 않은가 Codex 하네스에서 실행 불가능한 지시가 섞인다.
현재 테스트 상태

docs/skill-behavior-contract-tests.md 기준으로 11개 계약과 35개 시나리오가 정의되어 있다. 여기에는 기존 what 골든 레퍼런스와 이번 curated 10개 스킬이 포함된다.

운영 기준

새 스킬을 가져오거나 기존 스킬을 고칠 때는 아래 순서로 다룬다.

  1. 원본 의도와 Codex 매핑을 분리한다.
  2. codex/skills/<skill>/SKILL.md에 Codex-native 원본을 둔다.
  3. 질문, 중단, 도구 우선순위, 산출물을 행동 계약 시나리오로 추가한다.
  4. python3 tools/codex_skill_contract_verify.py --markdown-output docs/skill-behavior-contract-tests.md --json-output docs/skill-behavior-contract-tests.json를 실행한다.
  5. 필요할 때만 python3 tools/codex_curated_skill_sync.py --apply로 활성 스킬 루트에 동기화한다.
  6. 마지막에 python3 tools/codex_harness_verify.py --markdown-output docs/harness-verification-report.md --json-output docs/harness-verification-report.json로 전체 게이트를 확인한다.
주의

이 문서는 실제 headless 모델 응답을 채점한 결과가 아니라, 스킬 문서가 그런 행동을 충분히 강제하는지 분석한 결과다. 실제 모델 회귀 테스트가 필요하면 같은 시나리오를 Codex exec 기반 runner에 연결하는 다음 단계가 필요하다.
