2026-05-04 기준으로 Codex active harness 상세 검증은 FAIL 0을 유지하고 있다. 이 문서는 그 다음 단계로, 외부 GitHub 하네스 후보를 우리 기준에 맞춰 읽고 어떤 부분만 선별 흡수할지 판단하기 위한 자료다.

목차 진입 게이트 우선순위 후보별 판단 흡수 방식 다음 작업 출처

진입 게이트

외부 하네스는 우리의 Codex active harness를 보강하기 위한 참고 자료다. 그래서 외부 후보를 먼저 설치하거나 덮어쓰지 않는다.

19 상세 검증 PASS
2 문서화된 WARN
0 검증 FAIL
49 Codex-ready skills
판단 기준: 외부 후보는 설치 단위가 아니라 패턴 단위로 평가한다. 가져올 수 있는 것은 workflow, skill anatomy, 검증 규칙, 문서 템플릿, 상태 모델이다. 가져오지 않는 것은 인증, 세션, provider 설정, 홈 디렉터리 전체 덮어쓰기, 자동 install script다.
상세 notes: 후보별 3-pass 판단과 실제 import batch 제안은 docs/external-harness-import-notes.md에 기록한다.

후보별 판단

bkit-codex

우리 하네스와 가장 잘 맞는다. 이미 `source-command-pdca`, `think-*`, `pipeline-orchestrator`가 있으므로 bkit의 PDCA phase gate와 status model을 그대로 설치하지 말고 우리 skill 문법에 맞춰 재작성한다.

PDCA CE state model

agent-skills

senior engineering workflow의 표현이 좋다. 특히 검증 증거, anti-rationalization, review/test/security gate는 우리 `infra-audit`, `security-audit`, QA 계열을 다듬는 기준으로 쓸 수 있다.

quality gate review skill anatomy

github/spec-kit

외부 표준으로 가장 강하다. 우리 하네스의 project-bootstrap, source-command-pdca, what-ce 흐름에 constitution, specify, plan, tasks, analyze 단계를 대응시킨다.

SDD standard templates GitHub issues

cc-sdd

Kiro식 requirements-design-tasks를 Codex skills로 안정 지원한다는 점이 좋다. 다만 우리의 subagent 정책과 충돌하지 않도록 autonomous implementation 부분은 행동 지침으로만 가져온다.

EARS TDD independent review

oh-my-codex / OMX

community signal은 매우 크지만 runtime layer가 무겁다. Codex App에서는 default path가 아니므로 `$team`, `$ralph`, `.omx/` 같은 개념을 우리 방식의 skill/subagent/playbook으로 번역하는 선에서 시작한다.

runtime 보류 team UX 참고

ok-skills

범용 skill 모음이다. 이미 우리에게 있는 문서/PDF/PPT/XLSX/브라우저 계열과 중복이 많으므로 없는 영역만 골라 import 후보로 둔다.

portable skills docs lookup GitHub CI

codex-settings

설정 예시는 유용하지만 설치 방식은 `~/.codex`를 통째로 바꾸는 흐름이라 우리 자산 보존 원칙과 맞지 않는다. provider, prompt, skill discovery 설명만 참고한다.

no overwrite reference only

흡수 방식

  1. 외부 repo를 바로 설치하지 않고, 먼저 repo별 후보 노트를 만든다.
  2. 우리 기존 skill과 중복되는지 확인한다. 중복이면 외부 표현만 참고하고 skill을 새로 늘리지 않는다.
  3. 3-pass 검토를 적용한다: 원본 의도, Codex 매핑, 누락/충돌 검토.
  4. 반영 단위는 `SKILL.md`, `AGENTS.md` 규칙, 검증 스크립트, 문서 템플릿 중 하나로 작게 나눈다.
  5. 반영 뒤 `tools/codex_harness_verify.py`를 다시 실행하고 `FAIL: 0`을 유지한다.
중요: 외부 GitHub 후보의 install script, provider 설정, MCP server, 홈 디렉터리 overwrite는 바로 실행하지 않는다. 우리는 외부 하네스를 받아들이는 쪽이 아니라, 우리 하네스를 기준으로 외부 패턴을 컴파일하는 쪽이다.
